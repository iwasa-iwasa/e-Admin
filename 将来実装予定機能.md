## 🔄 部署異動承認フロー

### 概要
ユーザーが部署を変更する際に、管理者の承認を必要とする機能です。

### 背景
- 現在は、プロフィール編集画面で部署を変更すると即座に異動処理が実行される
- 実際の運用では、部署異動には承認プロセスが必要な場合がある
- 承認フローを導入することで、より厳格な部署管理が可能になる

### 実装内容（案）

#### 1. データベース設計
```sql
-- 部署異動申請テーブル
CREATE TABLE department_transfer_requests (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    from_department_id BIGINT UNSIGNED,
    to_department_id BIGINT UNSIGNED NOT NULL,
    status ENUM('pending', 'approved', 'rejected') DEFAULT 'pending',
    requested_by BIGINT UNSIGNED NOT NULL,
    reviewed_by BIGINT UNSIGNED,
    reason TEXT,
    reviewed_at TIMESTAMP NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (from_department_id) REFERENCES departments(id),
    FOREIGN KEY (to_department_id) REFERENCES departments(id),
    FOREIGN KEY (requested_by) REFERENCES users(id),
    FOREIGN KEY (reviewed_by) REFERENCES users(id)
);
```

#### 2. 承認フロー
1. **申請**: ユーザーまたは管理者が部署変更を申請
2. **承認待ち**: 全社管理者または部署管理者に通知
3. **承認/却下**: 管理者が申請を承認または却下
4. **実行**: 承認された場合、UserDepartmentTransferService を実行

#### 3. 権限設定
- **一般ユーザー**: 自分の部署変更を申請可能（管理者承認必須）
- **部署管理者**: 自部署メンバーの異動を申請可能（全社管理者承認必須）
- **全社管理者**: すべての異動を承認可能、即座に実行も可能

#### 4. UI/UX
- プロフィール編集画面に「部署変更申請」ボタンを追加
- 申請理由の入力フィールド
- 承認待ち一覧画面（管理者向け）
- 承認/却下ボタンとコメント機能

#### 5. 通知機能
- 申請時: 承認者に通知
- 承認時: 申請者に通知
- 却下時: 申請者に通知（理由付き）

### 実装優先度
