# 公開範囲についての変更

## 変更日
2024年

## 変更理由
各機能の目的を考慮すると、現在の公開範囲の選択肢には不合理な点がありました。
- カレンダー予定で部署カレンダーから全社公開する意味がない
- 共有機能（カレンダー、メモ、アンケート）で非公開にするなら個人リマインダーで十分
- ユーザーに不要な選択肢を提示することで混乱を招く

## 変更内容

### 1. カレンダー予定（Event）

#### 変更前
ユーザーが公開範囲を選択：
- public（全社公開）
- department（部署公開）
- private（非公開）
- custom（カスタム）

#### 変更後
カレンダーの種別から自動決定（ユーザーは選択不要）：
- **全社カレンダー** → 自動的に public（全社公開）
- **部署カレンダー** → 自動的に department（部署公開）
- **個人用** → 個人リマインダー機能を使用

#### 理由
- 部署カレンダーで作成した予定を全社公開する必要性がない
- 共有カレンダーで非公開にするなら個人リマインダーを使うべき
- カレンダーの種別で公開範囲は自明

---

### 2. 共有メモ（SharedNote）

#### 変更前
ユーザーが公開範囲を選択：
- public（全社公開）
- department（部署公開）
- private（非公開）
- custom（カスタム）

#### 変更後
以下の2つのみ：
- **department（部署内共有）**: 自部署のメンバーと共有
- **custom（カスタム共有）**: 特定のメンバーを指定して共有

**例外**: カレンダー予定に紐づくメモは予定の公開範囲を継承
- 全社カレンダーの予定に紐づく → public（全社公開）
- 部署カレンダーの予定に紐づく → department（部署公開）

#### 理由
- 「共有」メモなのに非公開（private）は矛盾
- 部署の内部情報を全社に公開する必要性が低い
- 全社向けのお知らせは全社カレンダーの予定経由で作成される

---

### 3. アンケート（Survey）

#### 変更前
ユーザーが公開範囲を選択：
- public（全社公開）
- department（部署公開）
- private（非公開）
- custom（カスタム）

#### 変更後
以下の3つのみ：
- **public（全社公開）**: 全社員が対象のアンケート
- **department（部署公開）**: 特定部署が対象のアンケート
- **custom（カスタム）**: 特定のメンバーが対象のアンケート

#### 理由
- アンケートを非公開（private）にする意味がない
- アンケートは回答者を集める必要がある

---

## 技術的な変更

### 1. バリデーションルールの変更

#### StoreEventRequest.php / UpdateEventRequest.php
```php
// 変更前
'visibility_type' => 'required|string|in:public,department,private,custom',

// 変更後
// visibility_typeフィールドを削除（自動決定）
```

#### StoreSharedNoteRequest.php / UpdateSharedNoteRequest.php
```php
// 変更前
'visibility_type' => ['required', 'string', 'in:public,department,private,custom'],

// 変更後
'visibility_type' => ['required', 'string', 'in:department,custom'],
```

#### StoreSurveyRequest.php / UpdateSurveyRequest.php
```php
// 変更前
'visibility_type' => ['required', 'string', 'in:public,department,custom,private'],

// 変更後
'visibility_type' => ['required', 'string', 'in:public,department,custom'],
```

---

### 2. コントローラーの変更

#### EventService.php
```php
// カレンダーの種別から公開範囲を自動決定
$calendar = Calendar::find($data['calendar_id']);

if ($calendar->owner_type === 'company') {
    $visibility_type = 'public';
    $owner_department_id = null;
} elseif ($calendar->owner_type === 'department') {
    $visibility_type = 'department';
    $owner_department_id = $calendar->owner_id;
}

$event = Event::create([
    // ...
    'visibility_type' => $visibility_type,
    'owner_department_id' => $owner_department_id,
]);
```

#### NoteController.php
```php
// カレンダー予定に紐づく場合は予定の公開範囲を継承
if ($validated['linked_event_id']) {
    $event = Event::find($validated['linked_event_id']);
    $visibility_type = $event->visibility_type;
    $owner_department_id = $event->owner_department_id;
} else {
    $visibility_type = $validated['visibility_type']; // department or custom
    $owner_department_id = $visibility_type === 'department' 
        ? Auth::user()->department_id 
        : null;
}
```

### 3. フロントエンドの変更

#### カレンダー予定作成/編集フォーム
- 公開範囲の選択UIを削除
- カレンダー選択時に公開範囲を自動表示（読み取り専用）

#### 共有メモ作成/編集フォーム
- 公開範囲の選択肢を「部署内共有」「カスタム共有」のみに変更
- カレンダー予定に紐づく場合は公開範囲を自動設定（読み取り専用）

#### アンケート作成/編集フォーム
- 公開範囲の選択肢から「非公開」を削除
- 「全社公開」「部署公開」「カスタム」のみ表示

## データベースの変更

### 不要になったデータ
以下の組み合わせは今後作成されない：
- カレンダー予定で `visibility_type = 'private'`
- カレンダー予定で `visibility_type = 'custom'`
- 共有メモで `visibility_type = 'private'`
- 共有メモで `visibility_type = 'public'`（予定に紐づく場合を除く）
- アンケートで `visibility_type = 'private'`

### 既存データの扱い
既存の不合理なデータは以下のように移行：
```sql
-- カレンダー予定: privateをdepartmentに変更
UPDATE events 
SET visibility_type = 'department', 
    owner_department_id = (SELECT department_id FROM users WHERE id = events.created_by)
WHERE visibility_type = 'private';

-- 共有メモ: privateをdepartmentに変更
UPDATE shared_notes 
SET visibility_type = 'department',
    owner_department_id = (SELECT department_id FROM users WHERE id = shared_notes.author_id)
WHERE visibility_type = 'private';

-- アンケート: privateをdepartmentに変更
UPDATE surveys 
SET visibility_type = 'department',
    owner_department_id = (SELECT department_id FROM users WHERE id = surveys.created_by)
WHERE visibility_type = 'private';
```

## 削除される機能

### 1. カレンダー予定の公開範囲選択UI
- 予定作成/編集フォームから公開範囲選択を削除
- カレンダー選択時に自動決定される公開範囲を表示（情報表示のみ）

### 2. 共有メモの全社公開・非公開オプション
- 「全社公開」オプションを削除（予定に紐づく場合を除く）
- 「非公開」オプションを削除

### 3. アンケートの非公開オプション
- 「非公開」オプションを削除

---

## ユーザーへの影響

### メリット
1. **選択肢の削減**: 不要な選択肢がなくなり、迷わない
2. **一貫性の向上**: カレンダーの種別と公開範囲が一致
3. **誤操作の防止**: 不合理な設定ができなくなる
4. **理解しやすさ**: 機能の目的が明確になる

### 注意点
1. **既存データ**: 不合理な設定の既存データは自動的に修正される
2. **操作の変更**: カレンダー予定では公開範囲を選択できなくなる
3. **共有メモ**: 全社公開したい場合は全社カレンダーの予定経由で作成

## 今後の拡張性

### 全社カレンダー承認フロー
部署の予定を全社に共有したい場合：
1. 部署カレンダーで予定を作成（部署内のみ表示）
2. 全社カレンダーへの追加を申請
3. 全社管理者が承認
4. 全社カレンダーに予定がコピーされる（全社公開）

この機能により、部署カレンダーから直接全社公開する必要がなくなります。

## 参考: 公開範囲の定義

| 値 | 名称 | 説明 |
|---|---|---|
| public | 全社公開 | 全社員が閲覧可能 |
| department | 部署公開 | 所属部署のメンバーのみ閲覧可能 |
| custom | カスタム | 指定したユーザーのみ閲覧可能 |
| ~~private~~ | ~~非公開~~ | ~~作成者のみ閲覧可能（削除）~~ |

## まとめ

この変更により、各機能の目的に沿った合理的な公開範囲設定が可能になります。
ユーザーは不要な選択肢に悩まされることなく、直感的に機能を使用できるようになります。
